🌟 什么是 Retrieval-Augmented Generation (RAG)？

RAG 是一种技术：在你提问时，大语言模型（LLM）会从外部文档中检索相关内容，再结合这些内容生成回答。

不同数据来源处理方式：
数据来源 使用 Loader 技术核心
PDF 文档 PyPDFLoader 按页提取文本与元数据
YouTube 音频 YoutubeAudioLoader + Whisper 转录视频音频为文字
网页内容 WebBaseLoader 从网页抓取文本
Notion 导出 NotionDirectoryLoader 批量读取 Markdown 格式的笔记内容

🧠 为什么要对文档进行切分？

大语言模型（LLM）有 上下文长度限制，比如 GPT-4 支持约 128k tokens 的上下文，但普通使用最多只能输入几千到几万 tokens。

因此：

  • 在将文档传入 LLM 进行问答之前，需要将大文档拆成若干小块（chunks）；
  • 这些小块应该尽量保持语义连续性，防止切断句子或逻辑结构；
  • 切分结果会用于后续的向量化 + 召回 + RAG问答。

在RAG（检索增强生成）流程中，文档分割的目的是：

  • 将大文档拆分为语义连贯的小块（chunks）
  • 确保分割后的文本能保留上下文信息
  • 适应LLM的上下文窗口限制（如GPT-4的token上限）

LangChain 提供了多个切分器（TextSplitter 类）：

类型 名称 特点
简单字符切分 CharacterTextSplitter 按字符长度切分（可设分隔符、重叠）
递归字符切分 RecursiveCharacterTextSplitter 优先按段落/句子等大粒度分隔符切分，不够再细分
Token 切分 TokenTextSplitter 按 token 数切分，适合控制上下文长度
Markdown 切分 MarkdownHeaderTextSplitter 利用 Markdown 标题结构进行层级化分割

看一个例子：

分割器对比

Splitter 切分依据 优点 缺点
CharacterTextSplitter 字符 + 分隔符 快速、简单 不考虑 token 数，易超限
RecursiveCharacterTextSplitter 多级分隔符 + 语义结构 保语义完整、结构清晰 仍然是字符级别，token 数可能不精确
TokenTextSplitter token 数 与模型 token 精确对齐 tokenizer 依赖，切分不一定保语义完整

CharacterTextSplitter VS RecursiveCharacterTextSplitter

特性 CharacterTextSplitter RecursiveCharacterTextSplitter
分隔符依赖 必须找到指定分隔符 可降级到字符级切割
默认分隔符 ""（空字符串） ["\n\n", "\n", " ", ""]（多级）
切割确定性 严格遵循分隔符 更灵活，保证一定有输出
适用场景 结构化文本（如代码、Markdown） 通用文本（尤其是未知格式的文本）
