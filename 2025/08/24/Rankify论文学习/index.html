Rankify论文学习

发表于2025-08-24|更新于2025-08-24
|阅读量:

Rankify摘要

近年来，信息检索（IR）取得了显著进展，涵盖了稀疏检索、稠密检索以及它们的混合方法。这些方法结合了稀疏检索器的精确性和稠密模型的语义理解能力，从而实现了更均衡和更稳健的检索系统。
重排序技术进一步提升了初始检索结果，确保最相关的文档排在前列。这类方法主要分为三类：

  1. Pointwise 重排序：将重排序视为回归或分类任务，为每个文档独立分配相关性分数。
  2. Pairwise 重排序：通过比较文档对并优化它们的相对顺序来改进排序，但有时会以牺牲前列结果为代价改善后列结果。
  3. Listwise 重排序：考虑整个文档列表，利用 LLMs 将查询与候选子集结合进行重排序。由于输入长度限制，这类方法通常采用滑动窗口策略，从后往前逐步优化排序。

检索增强生成（RAG） 将检索与生成结合，在知识密集型任务中展现出更强能力。RAG 不再完全依赖预训练知识，而是在推理过程中动态检索外部文档，并将其融入生成中。

然而，检索、重排序和RAG方法的复杂性和多样性带来了基准测试、复现和集成的巨大挑战。现有工具包（如 Pyserini、Rerankers、RankLLM）往往缺乏灵活性、实现方式僵化，并且需要大量预处理，因此不适合研究驱动的实验。此外，检索和重排序数据集分散在不同来源，增加了评估和比较的难度。

为了解决这些问题，我们提出了 Rankify —— 一个开源框架，将检索、重排序和RAG 统一在一个模块化、可扩展的生态系统中。Rankify 支持多样化的检索技术，集成了最新的重排序模型，并提供了精心整理的预检索数据集，以简化实验。它旨在最大化灵活性，使研究人员能够高效地构建、评估和扩展检索管道，同时确保基准测试的一致性。

本文主要贡献

  1. 精选的检索数据集和预计算嵌入
    • 提供了 40 个数据集，每个数据集包含每个查询对应的 1000 个预检索文档，覆盖 QA、对话、实体链接等多个领域。
    • 包含了 Wikipedia 和 MS MARCO 语料库的多种检索器预计算结果，消除了预处理开销。
  2. 多样化的检索器与重排序支持
    • 集成了稠密检索器（如 DPR、ANCE、BPR、ColBERT、BGE、Contriever）和稀疏检索器（如 BM25），以及 24 个最先进的重排序模型，支持无缝检索与排序实验。
  3. RAG 集成与评估工具
    • 通过将检索到的文档传递给 LLM，用于生成式任务的评估，打通了检索、重排序和RAG。
  4. 全面的评估工具
    • 提供了多样化的评估指标和工具，支持检索和问答的效果评估。
    • 附带了详尽的在线文档，并已在 PyPI 和 GitHub 上开源，方便研究人员和从业者使用。

2 相关工作

2.1 稀疏与稠密检索

传统的稀疏检索方法，如 BM25，依赖词项匹配和词频-逆文档频率（TF-IDF）等启发式方法来衡量文档的相关性。这类方法在需要精确匹配的场景中表现良好，但在处理同义词、语义变化时效果有限。

稠密检索方法利用神经网络将查询和文档编码到连续向量空间中，并通过相似度计算进行匹配。代表性方法包括 DPR（Dense Passage Retrieval）、ANCE 和 Contriever 等。这类方法在捕捉语义信息方面具有优势，但在需要高精度词项匹配的任务上，往往不如稀疏检索。

近年来，研究者提出了结合稀疏与稠密方法的 混合检索（Hybrid Retrieval）。这种方法兼顾稀疏检索的精确性与稠密检索的语义理解能力，在各种检索任务中展现了更稳健的性能。

2.2 重排序方法

重排序旨在在初步检索的基础上进一步提升结果的相关性。根据不同的优化粒度，重排序可分为三类：

  1. Pointwise 方法：将重排序问题建模为回归或分类任务，对每个候选文档单独打分。代表性方法包括 BERT-based rerankers。
  2. Pairwise 方法：直接优化文档对之间的相对顺序，通过学习偏好来提升排序效果。例如 RankNet。这类方法有时会通过改善尾部文档排序来牺牲头部质量。
  3. Listwise 方法：直接优化整个候选文档列表。近期方法利用 LLM（大型语言模型） 在给定查询和候选文档子集的情况下进行排序。然而，受限于上下文长度，通常需要采用 滑动窗口策略，从后往前逐步优化文档顺序。

2.3 检索增强生成（RAG）

RAG（Retrieval-Augmented Generation） 在知识密集型任务（如开放域问答和事实性生成）中展现了显著潜力。
它的核心思想是：在生成过程中动态检索外部文档，并将其与输入一起送入生成模型。

代表性工作包括：

  • REALM：在预训练过程中集成检索器
  • RAG（Lewis et al., 2020）：结合 DPR 检索与生成模型
  • Atlas（Izacard et al., 2022）：通过指令调优（instruction tuning）进一步提升性能

这些方法避免了仅依赖模型内部参数存储知识的局限性，使模型在面对知识更新和新领域时更具适应性。

2.4 工具包与基准

近年来，已经有一些工具包支持检索与重排序研究：

  • Pyserini：为稀疏检索提供了便捷接口，但在稠密检索与RAG方面功能有限。
  • Rerankers：主要关注重排序模型的评估，但缺乏与检索的无缝集成。
  • RankLLM：针对 LLM-based 重排序进行了探索，但在模块化和灵活性上仍有不足。

这些工具在特定方面提供了支持，但普遍存在以下问题：

  1. 灵活性不足，难以快速定制新实验；
  2. 缺乏统一的数据预处理与评估基准；
  3. 难以在检索、重排序与RAG之间实现无缝衔接。

因此，亟需一个综合性的工具，既能支持多样化的检索和重排序方法，又能覆盖 RAG 任务，同时具备模块化和可扩展性。

3 Rankify 框架

为了应对现有工具在灵活性、复现性和集成性上的不足，我们提出了 Rankify —— 一个模块化、可扩展的开源框架，用于 检索、重排序和检索增强生成（RAG） 的统一实验。

Rankify 的设计目标是：

  1. 模块化：每个组件（检索、重排序、生成、评估）都可以独立替换和扩展。
  2. 即插即用：研究者可以轻松组合不同检索器、重排序器和数据集，而无需繁琐的预处理。
  3. 复现性：通过统一的数据集和预计算嵌入，确保实验结果可复现。
  4. 可扩展性：支持新的模型和方法快速接入，适配未来的研究发展。

3.1 系统架构

Rankify 的整体架构由以下几个核心模块组成：

  • 检索器模块（Retriever Module）
    • 集成了稀疏检索（BM25 等）、稠密检索（DPR、Contriever、BGE 等）和混合检索方法。
    • 所有检索器均可通过统一接口调用。
  • 重排序模块（Re-ranker Module）
    • 提供了 24 种最先进的重排序模型，覆盖 Pointwise、Pairwise 和 Listwise 方法。
    • 支持 LLM-based 排序器，能够对长文档进行基于滑动窗口的处理。
  • RAG 模块（RAG Module）
    • 将检索与生成无缝结合，支持将检索到的候选文档传入生成模型，用于问答和知识密集型生成任务。
  • 数据与基准模块（Dataset & Benchmark Module）
    • 提供了 40 个数据集，每个查询包含 1000 个候选文档，涵盖 QA、对话、实体链接等任务。
    • 内置预计算嵌入，避免研究人员重复进行高开销的预处理。
  • 评估模块（Evaluation Module）
    • 提供了丰富的指标（NDCG、MRR、Recall@K 等）和分析工具。
    • 不仅支持检索与重排序的评估，还能评估 RAG 任务中的问答和生成质量。

3.2 工作流程

Rankify 的工作流程如下：

  1. 查询输入
    用户输入查询。
  2. 初始检索
    检索器从候选语料中返回一组候选文档。
  3. 重排序
    重排序模块根据不同策略（pointwise/pairwise/listwise）优化排序。
  4. RAG 生成（可选）
    如果任务是生成式（如问答），则将排序后的文档与查询一并输入 LLM，生成最终答案。
  5. 评估
    使用标准指标对检索、重排序和生成的结果进行评估。

3.3 开源与可用性

  • Rankify 提供了完善的 Python API，研究者可以用几行代码完成实验搭建。
  • 项目已开源至 GitHub，并在 PyPI 发布，便于安装与使用。
  • 文档和教程覆盖了常见任务场景，降低了上手门槛。

4 数据与检索模块

4.1 数据集整理

Rankify 提供了一个精心整理的数据集集合，涵盖 40 个不同的数据集，广泛应用于问答（QA）、对话、实体链接（Entity Linking）和信息检索（IR）等任务。

每个数据集都包含：

  • 查询（queries）：用户输入或任务相关的问题
  • 候选文档（candidate documents）：从语料库中检索的相关文档
  • 预检索结果（pre-retrieved results）：每个查询对应 1000 个候选文档，由多种检索器生成
  • 标注（annotations）：文档的相关性标签（用于监督训练或评估）

这些数据集覆盖了 开放域 QA（如 Natural Questions, TriviaQA）、段落检索（如 MS MARCO）、对话（如 TREC-CAR, Dialog datasets） 等场景，能够全面评估检索与重排序性能。

4.2 预计算嵌入

Rankify 内置了由多种检索模型生成的 预计算嵌入（pre-computed embeddings），包括：

  • 稠密模型（DPR、Contriever、BGE 等）
  • 稀疏模型（BM25、uniCOIL 等）
  • 混合检索（Hybrid Retrieval）

这样做的优势是：

  1. 避免重复开销：研究者无需在每次实验时重新编码整个语料库。
  2. 统一基准：保证不同实验和模型在相同的候选集上进行比较。
  3. 快速上手：研究者可以直接专注于检索方法和重排序策略的改进。

4.3 检索器模块

Rankify 集成了多种检索方法，涵盖 稀疏、稠密与混合 三类：

  • 稀疏检索（Sparse Retrieval）
    • 基于倒排索引和词频统计的方法，如 BM25。
    • 在需要精确匹配的任务上依然表现强劲。
  • 稠密检索（Dense Retrieval）
    • 使用神经网络将查询和文档映射到向量空间，并通过相似度度量（如内积、余弦相似度）进行匹配。
    • 支持的模型包括 DPR、ANCE、BPR、ColBERT、BGE、Contriever 等。
  • 混合检索（Hybrid Retrieval）
    • 将稀疏与稠密方法结合，例如将 BM25 的得分与稠密向量相似度加权融合。
    • 能够兼顾 词项精确匹配 与 语义理解，在多任务上取得更稳健的表现。

所有检索器通过 统一接口 封装，研究者只需简单配置即可在不同方法之间切换。

5 重排序模块

在 Rankify 中，重排序模块是核心组件之一，用于对初始检索返回的候选文档进行进一步优化排序。它集成了 24 种先进的重排序模型，覆盖 pointwise、pairwise 和 listwise 三种主要范式。

5.1 Pointwise 重排序

Pointwise 方法将重排序建模为 分类或回归任务，为每个候选文档单独分配一个相关性分数。

  • 代表模型：BERT-based rerankers（如 MonoBERT、MonoT5）。
  • 优点：实现简单，能够直接利用预训练语言模型。
  • 缺点：无法显式建模候选文档之间的相对顺序。

在 Rankify 中，用户只需加载相应的模型，即可对候选文档列表逐一打分，并基于得分进行排序。

5.2 Pairwise 重排序

Pairwise 方法通过比较文档对来优化排序，直接学习“文档 A 是否比文档 B 更相关”。

  • 代表模型：RankNet、LambdaMART 等。
  • 优点：能够学习候选文档之间的相对偏好。
  • 缺点：有时会过度优化尾部文档的排序，牺牲前列文档质量。

在 Rankify 中，pairwise 重排序器接受候选文档对作为输入，并通过优化成对偏好损失函数进行训练。

5.3 Listwise 重排序

Listwise 方法将候选文档列表整体作为输入，直接优化整个排序结果。

  • 代表模型：ListNet、ListMLE，以及 LLM-based re-rankers。
  • 优点：能够全局考虑文档之间的关系，提升排序一致性。
  • 缺点：受限于 LLM 的上下文长度，通常需要采用 滑动窗口策略：从候选列表的末尾向前逐步优化排序。

Rankify 实现了多种 listwise 方法，特别是 基于 LLM 的重排序器，能够处理长文档，并在开放域 QA 等任务中取得优异表现。

5.4 模块化设计

Rankify 的重排序模块具备以下特点：

  1. 统一接口：所有重排序方法均通过统一 API 调用，用户可以轻松切换不同范式。
  2. 可扩展性：支持研究人员添加新的重排序模型，只需实现标准接口即可。
  3. 组合灵活性：可以与任意检索器输出结合，实现 end-to-end 检索 + 重排序管道。
